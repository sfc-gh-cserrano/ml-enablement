Feature Store

Centralized Feature Management

The Snowflake Feature Store lets data scientists and ML engineers create, maintain, and use ML features entirely within Snowflake. At its core, a feature store is simply a Snowflake schema containing feature views — pipelines that transform raw data into reusable, governed features.

Feature views can be Snowflake-managed (backed by Dynamic Tables that auto-refresh incrementally) or external (backed by views maintained by tools like dbt). Both types are organized by entities — logical concepts like "customer" or "product" that connect features to their subjects.

Backend Implementation

Feature store = schema, feature view = dynamic table or view, entity = tag, feature = column. All standard Snowflake RBAC and governance applies automatically.

Why Snowflake Feature Store?

Traditional ML teams face common challenges: feature duplication across teams, training-serving skew, no governance over who accesses what features, and manual refresh pipelines. The Snowflake Feature Store addresses all of these by treating features as first-class managed objects.

🔒 Governance & Security

Data never leaves Snowflake. Fine-grained RBAC controls access to features. Audit who uses which features and when.

♻ Reusability

Standardize feature transformations in a central repository. Teams share features instead of rebuilding them, reducing duplication and inconsistency.

⚡ Auto-Refresh

Managed feature views (Dynamic Tables) auto-update incrementally as new source data arrives. No cron jobs, no orchestration code.

🕑 Point-in-Time Correctness

Backfill and PIT-correct features with ASOF JOIN. Prevents data leakage in time-series ML by ensuring features reflect only past information.

🔍 Discoverability

Snowsight UI for searching, browsing, and discovering features across your organization. Find features before creating new ones.

🔗 Lineage Integration

Automatic ML Lineage tracing from source tables to features to datasets to models. Know exactly what data trained which model.

Online vs Offline Feature Serving

The Feature Store supports two serving modes. Offline serving generates point-in-time correct datasets for model training and batch inference. Online serving provides low-latency feature lookups for real-time inference services, eliminating the need for callers to compute or pass all features.

Key Insight

Online serving enables models deployed as inference services (SPCS) to retrieve fresh features at prediction time without requiring all inputs from the caller. The model uses ML Lineage to know which features it needs and fetches them automatically.

Feature Versioning

Feature views support versioning to allow safe evolution of feature transformations without breaking downstream models. When you update a transformation (e.g., changing a window size or adding normalization), you register a new version. Existing models continue to use the old version until explicitly migrated.

from snowflake.ml.feature_store import FeatureStore, FeatureView, Entity

fs = FeatureStore(session=session, database="ML", schema="FEATURES")

# Define entity (the subject of features)
customer_entity = Entity(name="CUSTOMER", join_keys=["CUSTOMER_ID"])
fs.register_entity(customer_entity)

# Create a new version of an existing feature view
fv = FeatureView(
    name="customer_features",
    entities=[customer_entity],
    feature_df=new_transformation_df,
    refresh_freq="1 hour",
    desc="Updated aggregation window from 7d to 30d"
)

# Register as a new version - old version remains active
fs.register_feature_view(feature_view=fv, version="v2")

Version Coexistence

Multiple versions of a feature view can exist simultaneously. This enables A/B testing different feature definitions and gradual migration without downtime.

Serving Features

The Feature Store provides two primary APIs for consuming features. generate_training_set() creates point-in-time correct datasets for training by joining features to a "spine" DataFrame using temporal ASOF joins. retrieve_online_features() performs low-latency lookups for real-time inference.

Generate Training Sets

# Point-in-time correct training set generation
# spine_df contains entity keys + timestamps + labels
# Features are joined as-of the spine timestamp (no future leakage)
training_data = fs.generate_training_set(
    spine_df=label_df,
    features=[customer_fv, transaction_fv],
    spine_timestamp_col="EVENT_TIMESTAMP",
    spine_label_cols=["LABEL"]
)

# Result: DataFrame with all features joined correctly in time
training_data.to_pandas().head()

# Pass entity keys, get back latest feature values

Why Datasets?

In ML, data changes constantly. Without proper dataset management, you cannot reproduce experiments, trace what data trained a model, or guarantee consistency across team members. Snowflake Datasets solve this by providing immutable, versioned snapshots that are first-class Snowflake objects.

📷 Reproducibility

Immutable snapshots guarantee that any experiment can be exactly reproduced months or years later, even as source data changes.

🔗 Lineage Tracking

Automatic tracking from source tables through features to the dataset used for training. Know exactly what data produced which model.

⚡ Efficient Access

Optimized for distributed training with PyTorch, TensorFlow, and Snowpark ML via DataConnector. Stream data efficiently without loading everything into memory.

📦 Version Control

Multiple dataset versions within a single named object. Compare model performance across data versions to understand impact of data changes.

Snowflake Datasets

Datasets are immutable, versioned snapshots of data optimized for ML workloads. They bridge the gap between data engineering and model training. Once created, a dataset version is frozen — even if the underlying source data changes, your dataset remains constant. This is critical for regulatory compliance, model audits, and debugging production issues.

Immutability Guarantee

Once created, a dataset version cannot be modified or deleted (unless explicitly dropped). This ensures that model training results tied to a specific dataset version are always reproducible, even across teams and environments.

Why Experiment Tracking?

Data scientists typically run dozens or hundreds of training experiments before finding the best model. Without systematic tracking, teams lose track of which hyperparameters, data versions, and code produced which results. Snowflake's experiment tracking captures everything automatically so the best run can be confidently promoted to production.

📊 Compare Runs

Track metrics (accuracy, F1, loss), parameters (learning rate, depth), and artifacts across training runs. Identify the best model quantitatively.

📝 Automatic Logging

Autolog support for XGBoost, LightGBM, and sklearn with zero-code instrumentation. Just enable autolog and all metrics/params are captured.

🚀 Promote to Registry

Seamlessly promote the best experiment run to the Model Registry for deployment. One call bridges development and production.

Logging Metrics

Every experiment run captures parameters (inputs), metrics (outputs), and optionally the model artifact itself. You can log scalar values, dictionaries, or even confusion matrices. The tracking data is stored in Snowflake and visible in the Snowsight UI.

from snowflake.ml.experiment import Experiment

# Create or get an experiment
exp = Experiment(session=session, name="churn_model_experiment")

# Start a run and log metrics
with exp.start_run(run_name="xgboost_v1") as run:
    # Log hyperparameters
    run.log_param("learning_rate", 0.1)
    run.log_param("max_depth", 6)
    run.log_param("n_estimators", 500)

    # Train
    model.fit(X_train, y_train)
    preds = model.predict(X_test)

    # Log evaluation metrics
    run.log_metric("accuracy", accuracy_score(y_test, preds))
    run.log_metric("f1_score", f1_score(y_test, preds))
    run.log_metric("auc_roc", roc_auc_score(y_test, preds))

    # Save the model artifact with the run
    run.log_model(model)

Promoting an Experiment to the Registry

Once you've identified the best experiment run, promoting it to the Model Registry makes it available for production inference. The promotion process captures the model, its metrics, and its metadata as a versioned, governed object.

from snowflake.ml.registry import Registry
from snowflake.ml.model import task

reg = Registry(session=session, database_name="ML", schema_name="REGISTRY")

# Get the best run from experiment
best_run = exp.get_run("xgboost_v1")

# Log to the Model Registry for production use
mv = reg.log_model(
    model=best_run.get_model(),
    model_name="churn_predictor",
    version_name="v1",
    metrics=best_run.get_metrics(),
    sample_input_data=X_test,
    task=task.Task.TABULAR_BINARY_CLASSIFICATION,
    comment="Promoted from experiment: best F1=0.94"
)

What Happens at Promotion

The model is serialized, dependencies are captured, method signatures are inferred from sample_input_data, and the model becomes callable via SQL, Python, or REST API.

CPU & GPU Training

Snowflake ML leverages Compute Pools (Snowpark Container Services) for both CPU and GPU training workloads. Compute pools provide dedicated resources that scale independently of your virtual warehouses, enabling long-running training jobs without impacting query performance.

Instance FamilyTypeUse Case
CPU_X64_XS / SCPUSmall sklearn models, preprocessing
CPU_X64_M / LCPUXGBoost, LightGBM, larger data
HIGHMEM_X64_S / M / LHigh-Mem CPULarge datasets in-memory, pandas ops
GPU_NV_S1x GPUSingle-GPU PyTorch / TensorFlow
GPU_NV_M4x GPUMulti-GPU training, medium models
GPU_NV_L8x GPULarge-scale deep learning, LLM fine-tuning
 
-- Create a GPU compute pool for training
CREATE COMPUTE POOL gpu_training_pool
  MIN_NODES = 1
  MAX_NODES = 4
  INSTANCE_FAMILY = GPU_NV_M;

-- Pool auto-suspends when idle, auto-resumes on job submit

Cost Tip

Compute pools bill based on active node-hours. Set MAX_NODES to control costs, and use AUTO_SUSPEND to avoid paying for idle resources between jobs.

Distributed Training

For large datasets or complex models, single-node training is too slow. Snowflake provides purpose-built distributed APIs that handle data partitioning, worker communication, and fault tolerance automatically. Training can be 3-7x faster than equivalent open-source implementations.

XGBEstimator

Distributed XGBoost across compute nodes. Automatic data partitioning and AllReduce communication. Snowpark DataFrame in, trained model out.

Distributed

Multi-node LightGBM with built-in data sharding. Optimized for high-cardinality features and large datasets.

PyTorchDistributor

Multi-node, multi-GPU PyTorch with DistributedDataParallel. Launch any training script on compute pools with automatic NCCL setup.

Versioning Models

Models support unlimited versions, system aliases (DEFAULT, FIRST, LAST), custom aliases, and per-version metrics comparison. The default version is what gets called when you reference the model without specifying a version — enabling seamless model updates without changing downstream SQL.

# Log a new version of an existing model
mv_v2 = reg.log_model(
    model=improved_model,
    model_name="fraud_detector",
    version_name="v2",
    metrics={"accuracy": 0.97, "f1": 0.95}
)

# Set default version - downstream SQL automatically uses v2
model = reg.get_model("fraud_detector")
model.default = "v2"

# System aliases: DEFAULT, FIRST, LAST
latest = model.version("LAST")

# Compare metrics across all versions
for v in model.versions():
    print(f"{v.name}: {v.show_metrics()}")

SQL API Inference

The most powerful integration: invoke model methods directly in SQL queries. This means any SQL user, BI tool, Dynamic Table, or dbt model can run predictions without writing Python. The model appears as a callable object using the model_name!method_name() syntax.

-- Call model predict method inline in any SQL query
SELECT
    t.CUSTOMER_ID,
    t.AMOUNT,
    fraud_detector!predict(
        t.AMOUNT, t.MERCHANT_CATEGORY, t.HOUR_OF_DAY
    ) AS FRAUD_PROBABILITY
FROM TRANSACTIONS t
WHERE t.TRANSACTION_DATE = CURRENT_DATE;

-- Use in a Dynamic Table for continuous, streaming predictions
CREATE OR REPLACE DYNAMIC TABLE fraud_scores
    TARGET_LAG = '1 hour'
    WAREHOUSE = ML_WH
AS SELECT
    CUSTOMER_ID,
    fraud_detector!predict(*) AS score
FROM LIVE_TRANSACTIONS;

Dynamic Table Note

Models with VOLATILE volatility require full table refresh in Dynamic Tables. Use IMMUTABLE (default for built-in models) for incremental refresh support.

ML Jobs

ML Jobs let you run Python ML workloads on Snowflake Compute Pools from any development environment — VS Code, Jupyter, PyCharm, or CI/CD pipelines. Your code executes inside the Snowflake Container Runtime with GPU access, pre-installed ML packages, and automatic Snowpark session injection. No worksheets or notebooks required.

@remote Decorator

Wrap any Python function. It gets serialized, uploaded to a stage, and executed on your compute pool. Returns an MLJob handle for monitoring.

submit_file()

Submit standalone Python scripts with CLI arguments, custom pip dependencies, and environment variables. Great for existing training scripts.

submit_directory()

Submit multi-file projects with entrypoints, module imports, and directory structure. Ideal for production ML codebases.

from snowflake.ml.jobs import remote

@remote("MY_COMPUTE_POOL", stage_name="payload_stage")
def train_model(data_table: str):
    # This entire function runs on Snowflake compute
    # Session is auto-injected, GPU is available
    model = XGBClassifier()
    model.fit(load_data(data_table))
    return model

job = train_model("my_training_data")  # Non-blocking submit
print(job.status)   # PENDING, RUNNING, DONE, FAILED
model = job.result() # Blocks until done, returns trained model

Distributed ML Jobs

ML Jobs support multi-node execution via Ray. When you submit a job to a compute pool with multiple nodes, Snowflake automatically sets up a Ray cluster, distributes your workload, and provides a Ray Dashboard for real-time monitoring of tasks, memory, and GPU utilization.

from snowflake.ml.jobs import submit_file

job = submit_file(
    "distributed_train.py",
    "MULTI_NODE_POOL",  # Pool with MAX_NODES > 1
    stage_name="payload_stage",
    args=["--epochs", "100", "--batch-size", "256"],
    pip_requirements=["transformers", "accelerate"],
    external_access_integrations=["PYPI_EAI"]
)

# Real-time monitoring via Ray Dashboard
dashboard_url = job.get_ray_dashboard_url()
print(f"Dashboard: {dashboard_url}")

# Stream logs as they arrive
print(job.get_logs())

Ray Dashboard

Available in snowflake-ml-python 1.30+. Provides live visibility into cluster utilization, task scheduling, memory pressure, and worker health. Accessible via browser while the job is running.

Snowflake Gateway

Stable Endpoints, A/B Testing & High Availability

The Snowflake Gateway solves a critical production challenge: SPCS inference services are tightly coupled to their hostname, meaning service recreation breaks client integrations. The Gateway provides a permanent, stable URL that persists for the lifetime of the gateway object, decoupling clients from underlying service changes. This enables zero-downtime model upgrades, traffic splitting for A/B tests, and automatic failover for high availability.

🔗 Stable Endpoint

One permanent URL that never changes, regardless of how many times you recreate, upgrade, or replace the underlying inference service. Clients never need to update their integration.

🔀 A/B Testing & Canary Deploys

Split traffic across multiple model versions by percentage. Route 90% to production (V1) and 10% to candidate (V2). Gradually shift as confidence builds. Validate new models with real traffic before full rollout.

⚡ Automatic Failover

Gateway auto-redirects traffic away from unhealthy endpoints. If a service fails its readiness probe, is suspended, or is dropped, traffic flows to remaining healthy targets with proportional redistribution.

🚀 Blue-Green Deployments

Deploy V2 alongside V1, validate with small traffic %, then flip 100% to V2 in one ALTER command. Rollback instantly by shifting traffic back. Zero downtime, zero client changes.

Traffic Splitting Rules

  • Weights must sum to exactly 100
  • Up to 5 endpoints per gateway
  • Changes reflect within ~1 minute
  • Endpoints at 0% never receive failover traffic
  • Supports dataframe_records format for schema evolution

Failover Triggers

  • Service suspended (auto_resume=false)
  • Compute pool suspended
  • Readiness probe failure
  • Service dropped
  • Owner loses USAGE privilege on target

Schema Evolution Strategy

When V2 requires different input features than V1: (1) Update clients to send the superset of all features needed by both versions — Snowflake ignores extra columns. (2) Gradually shift traffic. (3) Once at 100% V2, clean up obsolete V1 features from client payloads.

How It Works

Create a model monitor object that points to your inference log table. The monitor automatically refreshes on schedule, computing metrics for each aggregation window (minimum 1 day). Logs, Events & Metrics

Drift Monitoring

Drift detection identifies when the distribution of input features or predictions has shifted from the training baseline. This is often the earliest signal that model accuracy is degrading — before ground truth labels become available. Snowflake supports PSI (Population Stability Index), KL Divergence, and JS Divergence for drift measurement.

-- Create a model monitor with baseline for drift detection
CREATE MODEL MONITOR churn_monitor WITH
    MODEL = ML.REGISTRY.CHURN_PREDICTOR
    VERSION = 'v2'
    SOURCE = ML.DATA.INFERENCE_LOG
    TIMESTAMP_COLUMN = INFERENCE_TS
    PREDICTION_SCORE_COLUMNS = (CHURN_PROBABILITY)
    ACTUAL_SCORE_COLUMNS = (ACTUAL_CHURN)
    ID_COLUMNS = (CUSTOMER_ID)
    AGGREGATION_WINDOW = '1 DAY'
    WAREHOUSE = ML_MONITOR_WH
    BASELINE = ML.DATA.TRAINING_BASELINE
    SEGMENT_COLUMNS = (CUSTOMER_TIER, REGION);

-- Query drift: which features are shifting?
SELECT *
FROM TABLE(MODEL_MONITOR_DRIFT_METRIC(
    'churn_monitor', 'PSI', 'INCOME',
    'DAY', '2026-04-01'::TIMESTAMP_NTZ, '2026-05-01'::TIMESTAMP_NTZ
));

Segmented Monitoring

Monitor drift per segment (e.g., by customer tier or region). A global model might look healthy while a specific segment is drifting significantly. Supports up to 5 segment columns with <25 unique values each.
