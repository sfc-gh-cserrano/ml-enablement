Streaming vs Non-Streaming

The agent:run API supports two response modes controlled by the stream boolean in the request body.

Streaming (default)

stream: true or omitted → SSE events via text/event-stream

  • Token-by-token typewriter effect
  • Delta events for thinking & text
  • Tool use/result events as they happen
  • Final response event aggregates all
event: response.text.delta
data: {"content_index": 0, "text": "Hello"}

Non-Streaming

stream: false → single JSON via application/json

  • Waits for full processing
  • Returns complete response object
  • Same schema as streaming response event
  • Simpler to consume, higher latency
{"role": "assistant",
 "content": [... // all content items ],
 "metadata": {"usage": ...}}
Tip: For non-streaming clients, you can subscribe only to the final response event in streaming mode — it is the logical aggregation of all prior events.

Streaming Event Taxonomy

Every SSE event has a named event: field and a data: JSON payload. Events fall into four categories.

Delta Events

token-by-token

Individual tokens streamed for typewriter rendering. Accumulate these to build the final block.

response.thinking.delta Reasoning token + signature
response.text.delta Answer token + elicitation flag
response.tool_result.analyst.delta Analyst: SQL, think, result_set, suggestions
▼ aggregates into

Content Item Events

aggregated blocks

Complete content blocks emitted once a section finishes streaming. These populate the final content[] array.

response.thinking Full reasoning text
response.text Full answer + annotations[]
response.text.annotation Individual citation object
response.tool_use Tool invocation request
response.tool_result Tool output + status
response.table ResultSet + title
response.chart Vega-Lite chart_spec

Status Events

progress & warnings

Lifecycle updates for progress indicators. Non-fatal warnings keep the stream alive.

response.status planning, executing_tool, proceeding_to_answer
response.tool_result.status Per-tool progress (e.g. "Executing SQL")
response.warning Non-fatal issue; stream continues

Terminal Events

stream end

Final events that close the stream. The response event is the aggregation of all prior events.

response Complete aggregated response (last event)
error Fatal error; code + message + request_id
metadata Thread message_id + run_id
Data Flow:  Deltas → Content Items → response (final)  — each category feeds into the next. Subscribe to deltas for real-time UX, or just the final response for simple clients.
Important: Your application must handle unknown event types gracefully. New events may be added in the future.

Thinking Events

Reasoning tokens let you show the agent's internal thought process. These appear before the final answer.

response.thinking.delta

Streamed one token at a time. Use for a typewriter effect on reasoning.

data: {
  "content_index": 0,
  "text": "The user is asking for a",
  "signature": "lorem ipsum"
}

response.thinking

Aggregated content item once thinking block is complete.

FieldTypeDescription
content_indexintegerPosition in content array

signaturestringSignature
event: response.thinking
data: {
  "content_index": 0,
  "text": "To answer your question I must...",
  "signature": "lorem ipsum"
}
UI Mapping: Render thinking tokens in a collapsible "Reasoning" panel with a dimmed or italic style. Accumulate response.thinking.delta tokens until response.thinking arrives with the final text.

Text & Annotation Events

Text events carry the agent's final answer. Annotations attach citations from Cortex Search results.

response.text.delta

Single answer token for typewriter rendering.

{
  "content_index": 0,
  "text": "Hello",
  "is_elicitation": false
}

response.text

Aggregated text block with annotations array.

{
  "content_index": 0,
  "text": "Lorem ipsum...",
  "annotations": [...],
  "is_elicitation": false
}

response.text.annotation

Individual annotation attached to text.

{
  "content_index": 0,
  "annotation_index": 0,
  "annotation": {
    "type": "cortex_search_citation",
    "doc_title": "Earnings Report",
    "text": "Revenue for 2025..."
  }
}

Annotation Schema (cortex_search_citation)

FieldDescription
typeAlways cortex_search_citation
indexCitation index in search results
search_result_idUnique ID for the search result
doc_idDocument UUID
doc_titleDocument title
textCited excerpt
UI Mapping:
• response.text.delta → Append tokens to answer panel
• response.text.annotation → Render inline citation markers (e.g. [1]) that expand on click
• is_elicitation: true → The agent is asking the user for more info — render as a prompt, not a final answer
• response.text → Final aggregated text; use for non-streaming display or as fallback

Streaming Analyst Flow

// 1. Thinking arrives first
event: response.tool_result.analyst.delta
data: {"delta": {"think": "Thinking..."}}

// 2. SQL query generated
event: response.tool_result.analyst.delta
data: {"delta": {"sql": "SELECT ..."}}

// 3. Query ID once execution starts
event: response.tool_result.analyst.delta
data: {"delta": {"query_id": "707787..."}}

// 4. Result set with data
event: response.tool_result.analyst.delta
data: {"delta": {"result_set": {...}}}

// 5. Final text response
event: response.tool_result.analyst.delta
data: {"delta": {"text": "The total..."}}

ResultSet Schema

Matches the Snowflake SQL API ResultSet.

{
  "statementHandle": "707787a0-...",
  "resultSetMetaData": {
    "numRows": 2,
    "format": "jsonv2",
    "rowType": [
      {"name":"col", "type":"VARCHAR"}
    ]
  },
  "data": [
    ["row1 col1", "row1 col2"],
    ["row2 col1", "row2 col2"]
  ]
}
Note: Currently the entire SQL comes in a single delta event, but may be streamed token-by-token in the future.

Tables & Charts

Rich content events deliver visual data components. Tables carry SQL result data; charts carry Vega-Lite specifications.

response.table

FieldDescription
content_indexPosition in content array
tool_use_idOriginating tool use ID
query_idSQL query ID that generated data
result_setFull ResultSet (metadata + 2D data array)
titleDisplay title for the table
event: response.table
data: {
  "tool_use_id": "toolu_123",
  "query_id": "6ac75378-...",
  "title": "Revenue by Month",
  "result_set": {
    "resultSetMetaData": {
      "rowType": [{"name":"MONTH",...}]
    },
    "data": [["Jan","10000"],...]
  }
}

response.chart

FieldDescription
content_indexPosition in content array
tool_use_idOriginating tool use ID
chart_specVega-Lite JSON spec (stringified)
event: response.chart
data: {
  "tool_use_id": "toolu_123",
  "chart_spec": "{\"$schema\":\"https://vega.github.io/schema/vega-lite/v5.json\",\"data\":{...},\"mark\":\"bar\"}"
}
Rendering: Parse chart_spec with JSON.parse() and pass to Vega-Lite's vegaEmbed() or a Vega-Lite React component.
Correlation: Use tool_use_id to link tables and charts back to the response.tool_use event that triggered them. This lets you group tool invocation + loading state + result in the same UI card.

Status, Warning & Error Events

These events provide lifecycle visibility and error handling.

response.status

Agent-level execution status updates.

FieldDescription
statusCurrent state enum
messageDescriptive message
{
  "status": "executing_tool",
  "message": "Executing tool `my_analyst_tool`"
}

Known statuses: planning, reasoning_agent_stop, proceeding_to_answer, executing_tool

response.warning

Non-fatal warnings. Stream continues.

FieldDescription
messageWarning text
{
  "message": "Unable to fetch tools from MCP server 'foo'. Response quality may be degraded."
}

error

Fatal error. Stream terminates.

FieldDescription
codeSnowflake error code
messageError description
request_idRequest UUID
{
  "code": "399504",
  "message": "Error during execution",
  "request_id": "61987ff6-..."
}
UI Mapping: Use response.status for progress indicators (e.g. "Planning...", "Executing tool..."). Show response.warning as dismissable banners. Show error as a red alert with the error code and request ID for debugging.

Non-Streaming Response (stream: false)

Set stream: false and Accept: application/json to receive a single JSON response. The body is identical to the streaming response event payload.

{
  "role": "assistant",
  "content": [
    {"type": "thinking", "thinking": {"text": "The user is asking about..."}},

    {"type": "tool_use", "tool_use": {
      "tool_use_id": "<id>", "type": "cortex_analyst_text_to_sql",
      "name": "semantic_view_a", "input": {"query": "..."}
    }},

    {"type": "tool_result", "tool_result": {
      "tool_use_id": "<id>", "status": "success",
      "content": [{"type":"json", "json": {
        "sql": "SELECT ...",
        "result_set": {"data": [["Electronics","3"]]},
        "text": "I can answer with the following SQL."
      }}]
    }},

    {"type": "text", "text": "Based on the data, there are 2 product types..."}
  ],
  "warnings": [],
  "metadata": {"usage": {"tokens_consumed": [...]}, "run_id": "123-456"}
}
Key Insight: The non-streaming content array preserves the full tool lifecycle: thinking → tool_use → tool_result → text. Iterate over it to render each block in order.

MessageContentItem — Content Type Map

The content array in the final response contains items of 6 types. Each maps to a specific UI component.

thinking

ThinkingContent

Agent reasoning text + signature

UI: Collapsible reasoning panel

text

String with optional annotations

The final answer with citations

UI: Markdown text + citation footnotes

tool_use

ToolUse

Tool invocation request with input

UI: Tool calling indicator

tool_result

ToolResult

Tool output: JSON or text content

UI: Expandable result card

table

TableContent

SQL result set with metadata + title

UI: Data table component

chart

ChartContent

Vega-Lite spec (stringified JSON)

UI: Vega-Lite chart render

Dispatch Pattern: 
content.forEach(item => {
  switch(item.type) {
    case 'thinking':  renderThinking(item.thinking);  break;
    case 'text':      renderText(item.text);          break;
    case 'tool_use':  renderToolUse(item.tool_use);   break;
    case 'tool_result': renderToolResult(item.tool_result); break;
    case 'table':    renderTable(item.table);         break;
    case 'chart':    renderChart(item.chart);         break;
  }
});
